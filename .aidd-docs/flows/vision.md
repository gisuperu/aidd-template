これは spec →（design）→ tasks → implement → review のループとは**別枠**の、大域の方針を決めるフェーズである。
成果物は `.aidd-docs/vision.md`（ビジョン）の新規作成または更新。**コードも spec.md も書かない。**

## ビジョンとは

- 「このプロジェクトはこういう形を目指す」という大域の合意。各 spec ループが参考にし、逸脱に気づくための基準になる**生きたドキュメント**（specs 履歴と違い、常に現行の合意を反映し、全ループから参照される)
- **具体になりすぎてはいけない**。書くのは: 目指すアーキテクチャの形（構成要素と境界、依存の方向）、設計原則と優先順位（迷ったときのデフォルト）、技術方針の大枠、目指さない方向
- 書かないもの: 画面・API の詳細、ファイル配置、ライブラリのバージョン、実装手順 —— これらは各 spec ループで決める内容

## 進め方

1. **知識の読み込み**: .aidd-docs/knowledge/ から `spec-writing` を読み込む。テーマに応じて `infra-architecture`（構成）、`project-structure`（コード配置の方針）、`security-practices`（認証・データ保護）も。
2. **現状把握**: 既存の `.aidd-docs/vision.md` とコード本体を読む（現行ループ以外の `.aidd-docs/specs/NNN-*` は読まない）。
3. **大域の対話**: 以下のスタイルで進める:
   - **一度に扱う論点は1つ**。論点を積み残しリストにして順に潰す
   - 選択肢は AskUserQuestion で提示し、各選択肢に**トレードオフ（得るもの/失うもの/後から変えるコスト）**を添える。推奨があれば理由付きで示す。ただし決めるのは人間
   - 人間の発言の曖昧な点は言い換えて確認する
   - **対話が具体に降りたら**（ファイル配置・バージョン・API 詳細など）、その決定は「/aidd-spec への引き継ぎメモ」に退避し、大域の論点に戻る
4. **ビジョンの更新**: 合意内容をもとに `.aidd-docs/vision.md` の更新案（差分の要約）を提示し、**人間の承認を得てから**保存する（`updated:` と変更履歴を更新）。
5. **spec ループへの接続**: 対話で出た具体の決定事項はビジョンに入れず、以下の形で次につなげる:
   - 「経緯・見送った選択肢」と「/aidd-spec への引き継ぎ事項」を `.aidd-docs/vision-notes/YYYY-MM-DD-テーマ.md` に検討メモとして保存するか確認する
   - 続けて仕様化する場合は、引き継ぎ事項を要件文の形に整えて「`/aidd-spec <この要件文>` で開始できます」と案内する（勝手に /aidd-spec を始めない）

## ルール

- `.aidd-docs/vision.md` を人間の承認なしに書き換えない（specループ中に乖離を見つけた場合も同じ。人間の判断を仰ぐ）
- 検討メモ（`.aidd-docs/vision-notes/`）は履歴。過去のメモを参照したい場合は人間に確認してから読む
- ビジョンが抽象的すぎて判断できない、と spec ループで問題になった場合のみ、具体度を一段上げる（それでもファイル名・バージョンまでは書かない）
