
対象範囲: （コマンド引数）

ドメイン・仕様の一元管理のためのコマンドである。前提となる考え方:

- **真実はコード＋テスト**。ドキュメントはそこから導出される「ビュー」にすぎない
- `docs/current-spec.md` は**手で編集しない生成物**。乖離したら直すのではなく再生成する（これにより二重管理・更新漏れが構造的に起きない）
- `.aidd-docs/specs/NNN-*`（ループ履歴）からは**生成しない**。履歴は古い可能性があるため、現在のコードとテストだけを情報源にする

## 手順

1. **抽出**: 対象範囲のコードとテストから現在の仕様を読み取る:
   - コード構造: モジュール/feature の一覧、各公開面（エクスポート・API エンドポイント・CLI コマンド等）、依存関係
   - 振る舞い: テスト名（「条件_期待結果」形式）とアサーション内容から、機能ごとの振る舞い・境界・異常系を抽出する
   - データ: 主要なエンティティ・スキーマ・状態遷移
2. **生成**: `docs/current-spec.md` を**全体を作り直す形で**再生成する:
   - frontmatter に `generated:`（日時）と `commit:`（生成時点の HEAD）を記録
   - 冒頭に「このファイルは /aidd-docs による生成物。手で編集しない。真実はコードとテスト」と明記
   - 構成: 機能一覧 → 機能ごとの振る舞い（テスト由来の箇条書き）→ データモデル → モジュール構成図
   - 読み手は「このプロジェクトに新しく入った人」。俯瞰できる粒度を保ち、コードの逐語的な転記はしない
3. **品質フィードバック**: 抽出中に「テスト名から振る舞いが読み取れない」箇所を見つけたら、生成物とは別にテスト名の改善候補として報告する（テスト自体は勝手に変更しない）
4. **報告**: 前回の生成物との主な差分（追加・変更・削除された仕様）を要約して人間に提示する。README への反映（概要の転記等）が有益と思われる場合は提案のみ行い、人間の承認を得てから反映する

## 実行タイミング（規定）

再生成が走るのは次の2つの場合**だけ**。それ以外のタイミングで Claude の判断で再生成しない:

1. **定期更新（1ループにつき1回）**: spec ループの /aidd-review 承認後（PR 作成前）に再生成し、そのループの PR に含める。これにより current-spec.md は常に「マージ済みの最新ループまでの姿」を表す
2. **人間の指摘による更新（随時）**: `/aidd-docs-fix <指摘内容>` で行う（原因をコード・テストと突き合わせて切り分けてから更新する。詳細は docs-fix.md）

ループ途中に current-spec.md が実装より古く見えるのは正常（ループ中の変更は承認後の定期更新で反映される）。参照時に不足を感じたら、current-spec.md ではなくコードとテストを直接読む。
