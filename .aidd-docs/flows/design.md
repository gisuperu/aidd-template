
対象仕様: （コマンド引数）

spec 承認後・タスク分解前に、実装の**構造（設計）**を具体化して人間のレビューを受けるフェーズである。
spec.md の frontmatter `design: required` のループでのみ実行する（条件付きフェーズ）。**このコマンドでは実装コードを一切書かない。**

## spec との責任分界（最初に理解すること）

- **spec.md = 振る舞い（what）**: 人間と対話して詰める要求。ユーザーから見た振る舞い・スコープ・受け入れ条件
- **design.md = 構造（how）**: それを実現するモジュール構成・データ構造・インターフェース・処理フロー。実装が進むと正当な理由で変わりうる
- **振る舞いの決定を design.md に書かない**。設計中に仕様の不足・矛盾に気づいたら、design 側で補って進めず、停止して spec.md の修正を人間に提案する
- 逆に、振る舞いが変わらない設計変更は design.md の修正＋再承認だけで済み、承認済みの spec.md を開き直さない（変更頻度の違うものを分けるのがこのフェーズの目的）

## 事前準備: 知識ファイルの読み込み

.aidd-docs/knowledge/ から**必要なものだけ**を読み込む（要件の洗い出しは spec で終わっているため、`spec-writing` `tdd-antipatterns` はこのフェーズでは読まない）:

- `ccsdd`（ワークフローの原則）— 必須
- `project-structure` — 新しいモジュール・ディレクトリの配置を決めるとき
- `infra-architecture` — コンテナ・ネットワーク構成を決めるとき
- `security-practices` — 認証・外部入力・秘密情報・ネットワーク公開に関わる構造を決めるとき

## 手順

1. **前提確認**: 対象の `.aidd-docs/specs/NNN-slug/spec.md` を読む。
   - 引数が省略された場合は `status: approved` かつ `design: required` かつ design.md 未作成の最新仕様を対象にする。
   - spec.md が `approved` でなければ**ここで停止**し、先に仕様レビューが必要である旨を人間に伝える。
   - spec.md が `design: skip` の場合は、設計フェーズが不要と判断された経緯を示して人間に確認する（人間が必要と判断したら spec.md の `design:` を `required` に更新してから進める）。
2. **現状調査**: 仕様に関係するコード本体と `.aidd-docs/vision.md` を読み、既存の構造・依存の方向を把握する（現行ループ以外の `.aidd-docs/specs/NNN-*` は読まない）。
3. **設計書の作成**: `.aidd-docs/specs/_template/design.md` をベースに `.aidd-docs/specs/NNN-slug/design.md` を作成する。
   - `status: draft` のまま
   - 設計項目は §D 番号付きで書く（後でタスクから参照するため）
   - 各節には採用した構造に加え、**主要な代替案と採らなかった理由**を短く書く（人間がレビューで判断できるように）
   - 該当しない節は「なし」と明記する（無理にすべての節を埋めない）
   - 設計が spec の「§7 環境変更・依存追加」にない依存を前提にしてはならない。新しい依存が必要な設計になったら、spec.md の §7 への追記を人間に提案してから進める
4. **ビジョンとの照合**: 設計がビジョンの構成・原則とズレる場合は、レビュー依頼の前に人間に確認する（ビジョン未記入ならスキップ）。
5. **レビュー依頼**: 最後に以下を人間に提示して**作業を止める**:
   - 設計の要点サマリ（どういう構造にしたか 3〜5 行）
   - 特にレビューしてほしい判断ポイント（トレードオフがある箇所、代替案と迷った箇所、仮決めした箇所）
   - 「この設計で問題なければ `/aidd-approve` を実行（または『承認』と返信）してください。修正点があれば指摘してください」と明示する

## 禁止事項

- 実装を始めること
- 振る舞い（仕様）の決定を design.md に書くこと（仕様の不足・矛盾は停止して spec.md の修正を人間に提案する）
- spec の §7 にない環境変更・依存追加を設計の前提にすること
- 人間の承認を待たずに `/aidd-tasks` へ進むこと
- `status` を勝手に `approved` にすること
- `status: draft` の design.md をコミットすること（コミットは承認後、`.aidd-docs/rules/implementation.md` の「コミットの粒度」に従いフェーズ0または実装のコミットに含める）
