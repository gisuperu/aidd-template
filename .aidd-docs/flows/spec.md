
要件: （コマンド引数）

以下の手順で仕様書を作成せよ。**このコマンドでは実装コードを一切書かない。**

## 事前準備: 知識ファイルの読み込み

.aidd-docs/knowledge/ から以下を読み込んでから作業する:

- `ccsdd`（ワークフローの原則）と `spec-writing`（要件チェックリスト・アンチパターン）— 必須
- `security-practices` — 認証・外部入力・秘密情報・ネットワーク公開に触れる要件のとき
- `infra-architecture` — コンテナ・ネットワーク構成を決める要件のとき
- `project-structure` — 新しいモジュール・ディレクトリ構成を決める要件のとき

## 手順

1. **番号の採番**: `.aidd-docs/specs/` 配下の既存ディレクトリ名を一覧して、次の連番 `NNN` と英小文字ケバブケースの短いスラッグを決める（例: `003-user-auth`）。
2. **ブランチ作成**: Git 管理下であればブランチ `spec/NNN-slug` を作成して切り替える（このループの作業はすべてこのブランチで行い、最終的に1つの PR になる）。
3. **現状調査**: 要件に関係する**コード本体・設定**と `.aidd-docs/vision.md`（ビジョン）を読み、現状の構造と制約を把握する。
   - **現行ループ以外の `.aidd-docs/specs/NNN-*` は読まない**（古い仕様は履歴であり、コードと食い違っている可能性がある。参照が必要なら人間の承認を得る）。
   - **ビジョンとの照合**: 要件がビジョンの方向性・原則とズレる場合は、仕様書を書く**前に**人間に確認する（今回の仕様を合わせるか、ビジョンを更新するか）。ビジョンが未記入ならスキップ。
4. **不明点の解消**: 仕様の方向性を左右する不明点があれば、仕様書を書く**前に** AskUserQuestion で人間に確認する。些細なことは自分で妥当なデフォルトを選び、仕様書の「未決事項」に判断理由とともに記載する。
5. **仕様書の作成**: `.aidd-docs/specs/_template/spec.md` をベースに `.aidd-docs/specs/NNN-slug/spec.md` を作成する。
   - `status: draft` のまま
   - 仕様詳細は §番号付きで書く（後でタスクから参照するため）
   - 「やらないこと（スコープ外）」を必ず明記する
   - **環境変更・依存追加を必ず明示する**: 新しいライブラリの追加（npm 等）、curl でのダウンロード、ツール/アプリケーションのインストールが必要なら、「環境変更・依存追加」セクションに**実行コマンド・バージョン・理由**を列挙する。これらは後で人間が1つずつ承認しながら実行する枠になる。不要な場合も「なし」と明記する
6. **レビュー依頼**: 最後に以下を人間に提示して**作業を止める**:
   - 仕様の要点サマリ（3〜5行）
   - **環境変更・依存追加の一覧**（人間が承認することになるコマンド。なければ「なし」）
   - 特にレビューしてほしい判断ポイント（トレードオフがある箇所、自分で仮決めした箇所）
   - 「この仕様で問題なければ `/aidd-approve` を実行（または『承認』と返信）してください。修正点があれば指摘してください」と明示する

## 禁止事項

- 実装を始めること
- 人間の承認を待たずに `/aidd-tasks` へ進むこと
- `status` を勝手に `approved` にすること
