
対象仕様: （コマンド引数）

以下の手順で実装せよ。ここが人間の手を離れて長時間任される段階なので、**進捗の可視化ルールと TDD ルールを厳守する**こと。

## 事前準備: 知識ファイルの読み込み

.aidd-docs/knowledge/ から以下を読み込んでから作業する:

- `tdd-antipatterns` — 必須（[RED]/[GREEN] の実行基準）
- `autonomous-work` — 必須（完了宣言の前の検証・報告・行動判断の規律）
- `security-practices` — 外部入力・認証・秘密情報・DB を扱うタスクがあるとき
- `project-structure` — 新規ファイル・ディレクトリを作るタスクがあるとき
- `infra-architecture` — Dockerfile・compose 等を書くタスクがあるとき
- `subagent-orchestration` — サブエージェントへ作業を委任・並列実装するとき

## 前提確認（満たさなければ停止して報告）

1. 対象の `spec.md` が `status: approved` であること
2. 対象の `tasks.md` が `status: approved` または `in-progress`（前回の続き）であること
3. Git 管理下ならブランチ `spec/NNN-slug` 上にいること（なければ作成して切り替える）
4. **現行ループ以外の `.aidd-docs/specs/NNN-*` は参照しない**。仕様は現行の spec.md、現状はコード本体から読む

## フェーズ0: 環境セットアップ【人間承認枠】

未完了のフェーズ0タスクがある場合、**自律実装に入る前に**ここを人間と対話しながら片付ける:

1. これから実行する環境変更コマンドの一覧を提示し、人間がまだ画面の前にいることを確認する
2. コマンドを**1つずつ**実行する。権限プロンプトは人間の承認に委ねる。**許可リストへの追加・サンドボックス回避・一括承認を求めてはならない**
3. 実行結果（バージョン等）を確認して `[x]` にする
4. フェーズ0がすべて `[x]` になったら、Git 管理下ならフェーズ0の成果 + tasks.md を **1コミット**にまとめる（`[NNN-slug] フェーズ0 環境セットアップ`）
5. 「ここから先は自律実装に入ります。進捗は tasks.md で確認できます」と伝えてから次へ進む

フェーズ0が「なし」または完了済みなら、この節はスキップしてよい。

## フェーズ1以降: 自律実装

1. tasks.md の `status` を `in-progress` に更新する。
2. 未完了タスクを**上から順に**、1タスクずつ以下のサイクルで処理する:

   ```
   a. tasks.md 更新: 該当タスクの行末に「 ← 🔄実装中」を追記、
      frontmatter の current: を更新して保存（コードを書く前に行う）
   b. spec.md の該当セクション（§X.X）を読み直す
   c. タスク種別に応じて実行する:
      - [RED]      仕様から分岐・境界値を網羅したテストを書き、
                   実行して「期待どおり失敗する」ことを確認する
      - [GREEN]    テストを通す最小限の実装をし、
                   対応テスト + 既存テストの全パスを確認する
      - [REFACTOR] テストを通したまま整理する
      テストを弱めて通すこと（ケース削除・アサーション緩和・skip）は禁止
   d. tasks.md 更新: [ ] → [x]、「🔄実装中」マーク削除、progress: 更新
      （タスクごとにはコミットしない。コミットは規定の2回のみ）
   ```

3. 全タスク完了後:
   - プロジェクト全体のテスト・Lint・ビルドを実行して通ることを確認する
   - tasks.md の `status` を `done` に更新する
   - Git 管理下なら実装 + tasks.md の最終状態を **1コミット**にまとめる（`[NNN-slug] 実装（フェーズ1〜N）`）
   - 「実装が完了しました。`/aidd-review` でレビュー資料を作成します」と伝えて `/aidd-review` の手順に進む

## 並列実装（任意。.aidd-docs/rules/implementation.md の並列規約に従う）

互いに独立な機能単位（フェーズ）が2つ以上残っている場合は、サブエージェント（作業者）への委任で並列化してよい。委任前に `subagent-orchestration`（委任判断・指示文の書き方・結果の検証）を読む。既定は逐次であり、独立性が明らかなときだけ使う:

1. **独立性の判定**: tasks.md のフェーズ間で (a) タスクの依存関係がない (b) 変更予定ファイルが重ならない (c) 担当する spec の§が別、をすべて確認する。1つでも怪しければ逐次
2. **委任**: 1機能単位（[RED]→[GREEN]→[REFACTOR] 通し）を1作業者に割り当てる。指示文には担当タスク・spec 該当§の全文・検証方法・変更してよいファイル範囲・禁止事項を自己完結で含める（作業者はルールファイルを読まない前提）。作業者は隔離された作業コピー（ワークツリー等）で作業させるのが望ましい
3. **進捗の可視化**: 主エージェントが着手時に各タスクへ `← 🔄実装中@A` `← 🔄実装中@B` と作業者タグ付きマーカーを付け、`current:` をカンマ区切りで更新する（人間は tasks.md だけで並列状況が分かること）
4. **統合**: 作業者の完了報告ごとに、主エージェントがテストを実行して検証 → タスク単位のコミットとして spec ブランチに取り込み → チェックボックスと `progress:` を更新する。統合は1単位ずつ直列に行う（同時マージしない）
5. **失敗時**: 作業者が仕様の不足・矛盾・3回試して解決しないエラーを報告したら、その単位を中断し、他の単位の統合を終えてから人間に報告する

## 実装中のルール（.aidd-docs/rules/implementation.md 準拠）

- 同時に「実装中」にできるタスクは作業者1体につき1つ（並列時は作業者タグ付きで複数可。並列規約に従う）
- 仕様にないことはやらない。仕様の不足・矛盾に気づいたら停止して人間に報告する
- 実装がビジョン（.aidd-docs/vision.md）の方針と乖離していると気づいたら停止して人間に報告する（spec側を直すかビジョンを更新するかは人間が判断。ビジョンを勝手に書き換えない）
- **フェーズ0にない環境変更（ライブラリ追加・ダウンロード・インストール）は実行しない**。必要になったら停止して人間に報告する
- 同じエラーに 3 回対処して解決しなければ停止して人間に報告する
- タスクの順序を勝手に変えない・飛ばさない（特に [RED] を飛ばして実装しない）
- 途中で中断・再開する場合も、tasks.md を見れば正確な状態が分かるように常に最新を保つ
