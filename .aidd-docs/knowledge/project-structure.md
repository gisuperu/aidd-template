# 結合度を下げるディレクトリ構成

言語・フレームワーク固有の慣習はプロジェクト側の規約（CLAUDE.md やプロジェクトスキル）が優先。ここは言語非依存の原則のみ。

## 原則

1. **機能（feature）単位でまとめる > 技術レイヤ単位でまとめる**
   - 1つの機能の変更が1つのディレクトリで完結するほど結合度は低い
2. **依存の方向は一方向**: `ドメイン(中心) ← アプリケーション ← インフラ/UI(外側)`。中心は外側を import しない
3. **共有コードは最小限**: 2箇所で使うからと安易に `shared/` に上げない。3箇所目が出てから昇格を検討
4. **機能間の直接 import 禁止**: 機能Aが機能Bの内部を import しない。必要なら B が公開するインターフェース（index 等の公開面）経由のみ

## 構成例（feature-based）

```
src/
├── features/
│   ├── auth/            # 1機能 = 1ディレクトリ
│   │   ├── index.*      # 公開面（他 feature はここ経由でのみ import 可）
│   │   ├── domain/      # ビジネスルール（外側に依存しない）
│   │   ├── service/     # ユースケース
│   │   ├── api/         # ルーティング・コントローラ
│   │   └── __tests__/   # この機能のテスト（実装と同居させる）
│   └── billing/
├── shared/              # 本当に横断的なものだけ（型・共通UI・低レベルユーティリティ）
└── infra/               # DB接続・外部APIクライアント・設定読み込み
```

## 判断テスト

新しいコードの置き場に迷ったら:

- 「この機能を丸ごと削除するとき、いくつのディレクトリを触るか」→ 1 が理想
- 「このファイルを変更すると、いくつの機能に影響が波及しうるか」→ 多いなら shared に置きすぎ
- 「循環 import は発生しないか」→ 発生するなら層の分け方が間違っている

## アンチパターン

| アンチパターン | 症状 | 直し方 |
|---|---|---|
| utils 肥大 | `utils/` に無関係な関数が積もる | 使う機能の近くに移す。横断的なものだけ残す |
| God module | 何でも知っている巨大モジュールに全機能が依存 | 責務ごとに分割し、依存を注入する |
| 双方向依存 | A→B と B→A の import が両方ある | 共通部分を抽出して両方がそれに依存する形へ |
| レイヤだけの分類 | `controllers/ models/ services/` 直下に全機能が混在 | 規模が小さいうちは可。機能が5個を超えたら feature 単位へ |
| テストの隔離 | テストが `tests/` 以下の遠い場所にあり実装と乖離 | 実装の隣に置く（プロジェクト規約が優先） |
| 深すぎる階層 | 5階層超のネスト | 3階層程度で保つ。深さより機能境界を優先 |

## 仕様書に書くとき

- spec の§技術方針（設計フェーズを使うループでは design.md の該当§D）に「新規ファイルの配置先」と「既存のどの公開面に依存するか」を書く。
- 既存構成がこの原則と異なる場合は**既存の慣習に合わせ**、乖離を /aidd-spec の未決事項で人間に確認する（勝手に大規模リストラしない）。
