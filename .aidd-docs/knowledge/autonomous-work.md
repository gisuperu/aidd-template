# 自律作業の知識: 検証・報告・行動の規律

## 完了の宣言は検証の後

コードを書いた・編集したこと自体は完了ではない。完了条件は「検証が通ったこと」。

- [ ] 「動くはず」で完了と言っていないか（テスト・ビルド・実際の実行で確認したか）
- [ ] 検証していない箇所を「未検証」と明示したか（曖昧に済ませていないか）
- [ ] 失敗した検証を、出力を添えてそのまま報告したか（成功したような要約にしていないか）
- [ ] テストが通らないとき、テストより先に自分の実装を疑ったか

検証していないことは「未検証」とはっきり書く。「〜のはずです」「問題ないと思われます」のように検証済みに読める曖昧な締め方はしない。テストが仕様と食い違うと判断した場合の扱いは rules/implementation.md 参照（勝手に直さず人間に報告）。

## 報告の書き方

- **結論を最初の1文に書く**: 何が起きたか・何が見つかったか・何が変わったか。経緯や根拠はその後に続ける
- **読み手は過程を見ていない前提で書く**: 作業中に自分が作った略語・コードネーム・番号（「案B」「パターン2」等）を説明なしに使わない
- **簡潔さより「読み返し不要」を優先する**: 短くするなら詳細を削るのであって、記号や矢印の連鎖（「A→B→失敗」等）に圧縮しない
- **途中でわかった重要事項は最終報告に再掲する**: 作業途中の発言は読まれない前提で書く
- **判断を仰ぐときは推奨を1つ添える**: 選択肢を並べるだけで終えない。推奨とその理由をセットで示す

## 行動の判断

- 行動に足る情報が揃ったら行動する。すでに確定した事実の再調査、決定済み事項の蒸し返し、採らない選択肢の長い列挙をしない
- 状態を変える操作（ファイル削除・上書き・設定変更・サービス再起動）の前に、いま持っている根拠がその操作を本当に支持しているか一度確認する。「既知の症状に似ている」だけで実行しない（似た症状で原因が違うことはよくある）
- 削除・上書きの前に対象の現物を見る。事前の説明・想定と実物が食い違っていたら、実行せず手を止めて報告する
- エラーが出ても即座に人間へ投げず、まず自分でエラーメッセージを読み、原因を調べ、再試行する（打ち切りの基準は rules/implementation.md の3回ルール参照）
- 「次は〜します」と宣言して止まらない。宣言したなら実行する。止まってよいのは、このプロジェクトの承認ゲート・レビュー待ち（rules/workflow.md）と、人間にしか決められない判断が必要なときだけ
- コード内のコメントはコードが示せない制約・理由のみ書く。「〜を修正」「この変更で正しく動く」のようなレビュアー向けの説明はコメントでなく報告に書く

## よくある逸脱と正しい形

| 逸脱 | なぜ起きるか | 正しい形 |
|---|---|---|
| 楽観的完了報告 | 検証を省いて「完了しました」/ 検証が面倒・通る自信がある | 検証してから報告する。未検証なら未検証と書く |
| 失敗の粉飾 | 通らないテストがあるのに「概ね完了」とまとめる / 悪い報告をしたくない | 失敗はそのまま出力付きで報告する。隠れた失敗は後で倍のコストになる |
| 宣言止まり | 「次に〜を確認します」で報告を終えて実際には確認しない / 区切りをつけたくなる | 宣言した作業を終えてから報告する |
| パターンマッチ操作 | 症状が既知のケースに似ているので同じ対処を即実行 / 過去の成功体験 | 根拠を1つ確認してから実行する |
| 記号圧縮された報告 | 矢印・略語だらけで読み手が解読を要する / 自分には自明 | 完全な文で書く。読み手は過程を見ていない |
