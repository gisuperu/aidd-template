# 実装レビューの観点

/aidd-review は「人間のレビュー時間を最小化する」ための資料作り。自分で先に以下の観点で洗い出し、問題は隠さず資料に載せる。

## 1. 仕様適合（最優先）

- [ ] spec の各§に対応する実装とテストが存在するか（対応表で1行ずつ確認）
- [ ] ビジョン（.aidd-docs/vision.md）の方針・原則と乖離していないか（乖離があるなら人間に確認済みか。ビジョン未記入ならスキップ）
- [ ] 仕様と**微妙に違う**実装がないか（エラーメッセージ、境界の含む/含まない、デフォルト値）
- [ ] スコープ外のこと・仕様にないことを実装していないか（善意の追加も⚠️として報告）
- [ ] 未決事項の仮決めが、仕様に書いたとおりに実装されているか

## 2. 正しさ

- [ ] 仕様の分岐表・境界値がすべてコードで扱われているか（elseの欠落、off-by-one）
- [ ] 異常系: 外部依存の失敗時にリソースリーク・中途半端な状態変化が起きないか
- [ ] 並行・重複実行で壊れないか（仕様に冪等性の記載がある場合）
- [ ] エラーの握りつぶし（空catch、無視される戻り値）がないか

## 3. テスト品質（tdd-antipatterns スキルの観点で）

- [ ] テストが仕様のケース一覧（[RED] タスクに列挙したもの）を網羅しているか
- [ ] トートロジー・アサーション欠如・モック過多になっていないか
- [ ] 実装中にテストを弱めた形跡がないか（skip、コメントアウト、期待値の書き換え）

## 4. セキュリティ（security-practices スキルの観点で）

- [ ] 「禁止」事項に該当する実装がないか（秘密のハードコード、文字列連結クエリ、平文パスワード等）
- [ ] 採用した「一時回避」がすべて既知の制限に返済条件付きで記録されているか

## 5. 保守性

- [ ] 新規ファイルの配置が既存構成・project-structure の原則に沿っているか（双方向依存を作っていないか）
- [ ] dead code・デバッグ出力・TODO の置き忘れがないか
- [ ] ループ限定の表記（spec §番号など）がコード・テストのコメントに残っていないか（specs は履歴になるためコード中の§参照はノイズ）
- [ ] tasks.md・spec.md の最終状態が実態と一致しているか（progress、変更履歴）

## 資料への載せ方

- 問題は**重大度順**（仕様違反・セキュリティ > 正しさ > テスト品質 > 保守性）に並べる。
- 各指摘に `ファイルパス:行番号` を付ける。
- **自分の実装の問題を発見したら、隠さず載せて対処案を添える**（その場で直せる軽微なものは直してから「修正済み」と報告してよい）。
- 「特にレビューしてほしい箇所」には、上のチェックで白黒つけられなかったもの・設計判断を載せる。
