# 実装ルール

## タスク進行の可視化（最重要）

人間は実装中、**tasks.md だけを見て進行状況を把握する**。以下を厳守:

### チェックボックスの状態表記

```markdown
- [ ] 1.1 未着手のタスク (spec §2.1)
- [ ] 1.2 実装中のタスク (spec §2.2) ← 🔄実装中
- [x] 1.3 完了したタスク (spec §2.3)
```

- **着手時**: 行末に ` ← 🔄実装中` を追記し、frontmatter の `current:` を更新して**すぐ保存**する。実装コードを書き始める前に行う。
- **完了時**: `[ ]` を `[x]` にし、` ← 🔄実装中` を削除し、`progress:` を更新する。
- 同時に「実装中」にしてよいタスクは **作業者（エージェント）1体につき1つ**。並列実装時は後述の並列規約に従い、`← 🔄実装中@A` のように作業者タグを付けて複数可（`current:` はカンマ区切り。例: `2.1, 3.1`）。

### タスクの完了条件

タスクを `[x]` にしてよいのは以下をすべて満たしたときのみ:

1. タスクに書かれた「検証」項目を実際に実行して通った（テスト・ビルド・動作確認）
2. tasks.md のチェックボックスを更新した

### コミットの粒度（人間の承認プロンプトを増やさない）

タスクごとにはコミットせず、Git 管理下では次の2回にまとめる:

- **フェーズ0完了時に1コミット**: `[NNN-slug] フェーズ0 環境セットアップ`（人間が同席しているうちに確定させる）
- **全タスク完了時に1コミット**: 実装 + tasks.md の最終状態を `[NNN-slug] 実装（フェーズ1〜N）`
- 例外: 実装を中断して人間に報告する場合は、その時点までの成果を1コミットに残してよい（作業消失を防ぐため）

## ファイル編集の方法

- ファイルの作成・変更は**エージェントの編集ツール（Read / Edit / Write 相当）で直接行う**。Python・sed・awk・ヒアドキュメント等のスクリプトやワンライナーでファイルを書き換えない（人間が差分を追えず、エスケープ・置換範囲の事故も起きやすい）。
- 対象ファイルが多くてもツールでの編集を繰り返す。スクリプト化したくなるほど機械的な一括変更は、実行前に人間に方法を確認する。
- ファイル内容の書き換えを伴わない操作（`git mv` によるリネーム、ビルド・テストの実行など）は通常どおりコマンドでよい。

## テスト駆動（TDD）ルール

タスクは機能単位ごとに `[RED] → [GREEN] → [REFACTOR]` の順で並んでいる。実装の粗（分岐の考慮もれ・境界条件）を防ぐための仕組みなので厳守する:

- **[RED] テスト作成タスク**: 仕様（§X.X）から正常系・異常系・分岐・境界値のケースを洗い出してテストを書く。完了条件は「テストを実行して**期待どおり失敗する**ことを確認した」こと。
- **[GREEN] 実装タスク**: 対応する [RED] のテストを通す最小限の実装をする。完了条件は「対応テスト + 既存テストがすべて通る」こと。
- **[REFACTOR] タスク**: テストを通したまま整理する。完了条件は「全テストが通ったまま」であること。
- **テストを弱めて通すことは禁止**（ケース削除・アサーション緩和・skip 化）。テストが仕様と食い違っていると判断した場合は、勝手に直さず人間に報告する。
- [RED] を飛ばして実装しない。順序の入れ替えも禁止。

## 環境変更タスク（フェーズ0【人間承認枠】）

ライブラリ追加（`npm install` 等）、`curl` によるダウンロード、ツール/アプリケーションのインストールなど**環境を変える操作**は、自律実装とは別枠で扱う:

- 実行してよいのは、spec.md の「環境変更・依存追加」セクションに明記され、tasks.md の**フェーズ0【人間承認枠】に載っているコマンドのみ**。
- フェーズ0は人間が同席している状態で実行する。コマンドは1つずつ実行し、**権限プロンプトは人間に承認させる**（許可リストへの追加・サンドバイパス・自動承認の要求は禁止）。
- フェーズ0がすべて完了するまで、フェーズ1以降の自律実装に入らない。
- 実装中に新しい依存が必要だと気づいた場合は、**その場でインストールせず**、作業を止めて人間に報告する（承認されたら spec とフェーズ0に追記してから実行する）。

## 仕様との対応

- 各タスクは spec.md のセクション番号（§X.X）を参照している。実装時はその仕様セクションを読み直してから書く。
- **spec の§番号をコード・テストのコメントに書き残さない。** §参照は tasks.md（ループの記録）にのみ書く。コードはループより長生きし、specs は履歴として参照禁止になるため、コード中の§番号は参照先のないノイズになる。コメントには仕様の「内容」を書く（例:「ファイルが存在しない場合は空リストを返す」）。
- **仕様に書かれていないことを実装しない。** 実装中に仕様の不足・矛盾・変更の必要に気づいたら:
  1. 現在のタスクを「実装中」のまま作業を止める
  2. 何が問題か・どんな選択肢があるかを人間に報告する
  3. 人間の判断を受けて spec.md を更新（変更履歴に追記）してから再開する

## 並列実装（サブエージェント）の規約

実装を担当するエージェントは、独立した作業をサブエージェント（作業者）に委任して並列化してよい。ただし以下を厳守する:

1. **並列化してよい単位は「互いに独立な機能単位（フェーズ）」のみ**: タスク間に依存がなく、変更するファイルが重ならず、spec の別々の§を担当するもの。独立性の判断に迷ったら逐次で実行する（既定は逐次。並列化は独立単位が2つ以上あり利得が明らかなときだけ）
2. **TDD の順序は各機能単位の中で厳守**: [RED]→[GREEN]→[REFACTOR] を1つの作業者が通しで担当する。単位の内側を並列化しない
3. **人間との接点と状態ファイルはオーケストレータ（主エージェント）専任**: tasks.md の更新、人間への確認・報告、フェーズ0（環境変更）、承認関連、spec ブランチへのコミット/マージは主エージェントのみが行う。**作業者に tasks.md を触らせない**
4. **作業者への指示は自己完結させる**: 作業者はこのルールファイルを読んでいない前提で、担当タスクと spec 該当§の内容・検証方法・変更してよいファイル範囲・禁止事項（テストを弱めない／環境変更しない／担当範囲外のファイルを変更しない／履歴 specs を参照しない／ファイル変更は編集ツールで行いスクリプトで書き換えない）を指示文に含める
5. **統合は検証してから**: 作業者の成果は主エージェントがテストを実行して確認したうえで作業ツリーに取り込む。コミットは通常規定どおり「全タスク完了時の実装1コミット」にまとめる
6. **作業者が詰まったら人間へ**: 作業者が仕様の不足・矛盾・解決できないエラーに当たったら、その機能単位を中断扱いにして主エージェントが人間に報告する（作業者に判断させない）
7. **読み取り専用の並列化は自由**: コードベース調査・呼び出し元の洗い出し・テスト実行などの読み取り専用作業の並列化は、実装に限らずどのフェーズでも制約なく使ってよい

委任判断・指示文の書き方・結果検証のチェックリストは `.aidd-docs/knowledge/subagent-orchestration.md` にある。

## 実装中に問題が起きたとき

- テストが通らない・エラーが解決できない場合、同じアプローチを3回試して駄目なら人間に状況を報告する。
- 先のタスクを飛ばして進めない。順序を変えたい場合は理由を添えて人間に確認する。
