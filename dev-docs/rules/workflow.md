# ワークフロールール

## 全体フロー

```
（別枠・随時）/design ── ビジョン（dev-docs/vision.md）を対話で固める・更新する
                        ※ ループの外側。具体の決定は「/spec への引き継ぎ」として渡す
（別枠・随時）/check ── 人間が直接行った細かい修正を Claude がレビューする
                        ※ spec ループを使うほどでもない変更向け。spec/tasks は作らない
（補助）/approve ── 承認ゲートの通過（人間が実行） ／ /status ── 進捗とビジョン実現度の一覧
（補助）/docs ── 現在仕様ドキュメントの再生成 ／ /docs-fix ── 現在仕様への指摘対応

人間: 要件を伝える
  ↓
Claude: /spec ── 仕様書ドラフト作成（dev-docs/specs/NNN-slug/spec.md）
  ↓
人間: 仕様レビュー ──修正指示──→ Claudeが仕様を修正（ループ）
  ↓ 「承認」
Claude: /tasks ── タスクリスト作成（dev-docs/specs/NNN-slug/tasks.md）
  ↓
人間: タスクレビュー ──修正指示──→ Claudeがタスクを修正（ループ）
  ↓ 「承認」
Claude: /implement ── タスクを順に実装（チェックボックスで進捗可視化）
  ↓
Claude: /review ── 実装レビュー資料を作成
  ↓
人間: 実装レビュー ──修正指示──→ Claudeが修正 → 再度 /review（ループ）
  ↓ 「承認」
完了（spec/tasks の status を done に更新）
```

## ステータス管理

spec.md / tasks.md の frontmatter にある `status` が唯一の真実:

| status | 意味 |
|---|---|
| `draft` | Claude が作成した。人間レビュー待ち |
| `approved` | 人間が承認した。次の段階に進んでよい |
| `in-progress` | 実装中（tasks.md のみ） |
| `done` | 実装完了・最終承認済み |

### 承認の扱い

- `status` を `approved` / `done` に変更してよいのは、**人間が明示的に承認の意思を示した場合のみ**。標準の経路は **`/approve` コマンド**（何を承認するか確認表示してから1段階だけ進める）。チャットでの明示承認（「承認」「OK、進めて」「LGTM」など）も同等に扱う。
- 承認時は frontmatter の `approved:` に日付を記録する。
- 曖昧な返事（「ふーん」「なるほど」など）は承認ではない。判断に迷ったら承認か確認する。
- 人間から修正指示があった場合は、該当ファイルを修正して再度レビューを依頼する。`status` は `draft` のまま。

## 仕様書の配置と「1ループ = 1ブランチ = 1PR」

- 1機能 = 1ディレクトリ: `dev-docs/specs/NNN-短いスラッグ/`（NNN は 001 からの連番）
- 中身: `spec.md`（仕様書）と `tasks.md`（タスクリスト）
- テンプレート: `dev-docs/specs/_template/` をコピーして使う
- **spec 作成から review 完了までの1ループが1つの PR の単位**。Git 管理下では:
  - `/spec` 開始時にブランチ `spec/NNN-slug` を作成し、以降このループの作業（spec.md / tasks.md / 実装）はすべてこのブランチにコミットする
  - `/review` で人間が承認したら PR の作成を提案する（spec.md へのリンクとレビュー資料を PR 本文に含める）

## ビジョン（dev-docs/vision.md）と spec ループの関係

`dev-docs/vision.md` は /design で固める「目指す形」の大域合意。**生きたドキュメント**であり、specs 履歴と違って**全ループから常に参照してよい**（現状把握は「コード本体 + ビジョン」から行う）。

- **/spec**: 要件をビジョンと照合し、仕様書に「ビジョンとの整合」を1〜2行記載する。ズレる場合は仕様書を書く前に人間に確認する
- **/implement /review /check**: 作業中にビジョンとの乖離に気づいたら人間に確認する
- ズレたときの人間の判断は2択: **(a) 今回の spec 側をビジョンに合わせて直す** / **(b) ビジョン自体を更新する**（/design で対話するか、軽微なら直接更新案を提示して承認を得る）
- ビジョンは具体になりすぎない抽象度（構成要素・原則・優先順位）を保つ。具体の判断基準が必要なら spec に書く
- ビジョンが未記入の間は整合チェックをスキップしてよい
- **実現度の記録**: ループ完了（/review 承認）ごとに、vision.md の「実現の記録」へ「どのビジョン項目にどれだけ近づいたか」を1行追記する。この欄は事実の記録なので追記に承認は不要（/status が項目別に集計し、実現度として可視化する）
- **Claude が人間の承認なしにビジョン本文（方針）を書き換えることは禁止**（「実現の記録」への追記のみ例外）

## ドキュメントの一元管理（dev-docs/current-spec.md と /docs）

ドメイン・仕様が複雑になっても二重管理が起きないよう、**真実はコードとテスト**に置く:

- 振る舞いの正確な仕様 = テスト（[RED] タスクで「条件_期待結果」形式の名前を付けるのはこのため）
- 俯瞰用の「現在の仕様」ドキュメント = `dev-docs/current-spec.md`。これは **/docs がコードとテストから再生成する生成物**であり、手で編集しない。乖離したら直すのではなく再生成する
- **更新タイミングは規定制**: (1) 各 spec ループの /review 承認後に1回（PR に含める）、(2) 人間が指摘したとき（抽出ミスなら再生成、コード側の問題なら /check・/spec へ誘導）。それ以外で Claude の判断により再生成しない
- `dev-docs/specs/`（履歴）や `dev-docs/current-spec.md` を仕様の根拠にしない。判断の根拠は常にコード・テスト・現行ループの spec.md
- `docs/` ディレクトリは**プロジェクトのユーザー向けドキュメント領域**として空けてある。テンプレートの開発者向け資料はすべて `dev-docs/` に置く

## 過去の仕様書（履歴）の扱い

`dev-docs/specs/` にはループごとのフォルダが履歴として蓄積される。**履歴であるがゆえに、古い仕様は現在のコードベースと食い違っている可能性がある。**

- **現行ループのフォルダ以外の `dev-docs/specs/NNN-*` を、人間の承認なしに参照・参考にしてはならない。** 検索（Grep/Glob 等）の対象からも除外する。
- 現状把握・調査は必ず**コード本体**から行う。コードが唯一の真実。
- 過去の仕様を参照する必要があると判断した場合（例: 人間が過去の機能名に言及した）は、どのフォルダをなぜ参照したいかを伝えて人間の承認を得てから読む。

## レビュー待ちで停止するルール

`/spec` `/tasks` `/review` の各コマンドの最後では、**必ず作業を止めて人間のレビューを待つ**。
「続けて実装します」のように勝手に次の段階へ進んではいけない。
レビューを依頼するときは、レビューしてほしいポイント（特に判断が分かれる箇所）を箇条書きで示すこと。
