
変更の意図: （コマンド引数）

これは spec → tasks → implement → review のループとは**別枠**の軽量フローである。役割が普段と逆になり、**人間が書いた変更を Claude がレビューする**。レビューが目的なので、**人間の依頼なしにコードを書き換えない**。

## 対象になる変更

typo・文言修正、小さなバグ修正、設定値の調整、軽微なリファクタなど、仕様を書くほどでもない変更。
レビュー中に「これは振る舞い・仕様が変わる規模だ」と気づいたら、その旨を伝えて `/aidd-spec` ループを提案する（レビュー自体は続けてよい）。

## 手順

1. **知識の読み込み**: dev-docs/knowledge/ から `review-perspectives` を読み込む。変更内容に応じて `security-practices`（入力・認証・秘密情報まわり）、`tdd-antipatterns`（テストの変更を含むとき）も。
2. **対象の特定**: Git 管理下なら未コミットの変更（staged + unstaged）を diff で確認する。引数やチャットで対象の指定があればそれに従う。対象が特定できなければ人間に確認する。
3. **意図の把握**: 変更の意図が diff から読み取れない場合のみ、人間に質問する（このフローでは人間が画面の前にいる前提でよい）。
4. **レビュー**: 軽量版の観点で確認する:
   - **正しさ**: 意図どおりか。境界・異常系・off-by-one。変更箇所の**呼び出し元への影響**（grep で波及先を確認）
   - **セキュリティ**: security-practices の「禁止」事項に該当しないか
   - **テスト**: 既存テストが通るか実行する。振る舞いが変わる修正ならテストの追加・更新が必要かを指摘する
   - **整合**: ビジョン（dev-docs/vision.md）の方針と矛盾しないか（現行ループ以外の specs 履歴とは照合しない。コードとビジョンが基準）
   - **置き忘れ**: デバッグ出力・コメントアウト・無関係な差分の混入
5. **報告**: 以下を提示して**作業を止める**:
   - 判定: **LGTM**（そのままコミット可）/ **要修正**（指摘あり）
   - 指摘は重大度順に `ファイルパス:行番号` 付きで、修正案を添える
   - テストの実行結果
   - LGTM の場合はコミットメッセージ案を添える

## 指摘後の流れ

- 修正するのは基本的に人間。人間が「直して」と言った指摘のみ Claude が修正し、修正した箇所を再確認して報告する
- 人間が修正した場合は、再度 `/aidd-check` で差分だけ再レビューする
- このフローでは spec.md / tasks.md は作らない・更新しない（specs はループの記録であり、軽量フローの変更は Git 履歴だけで十分）
