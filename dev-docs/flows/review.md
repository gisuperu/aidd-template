
対象仕様: （コマンド引数）

実装完了後の人間レビューのための資料を作成せよ。**新しい機能追加はしない。**

## 事前準備: 知識ファイルの読み込み

dev-docs/knowledge/ から `review-perspectives`（レビュー観点チェックリスト）を読み込んでから作業する。セキュリティに関わる変更を含む場合は `security-practices` も読み込む。

## 手順

1. 対象の `spec.md` / `tasks.md` と、実装の差分（Git 管理下なら `git diff` / コミットログ、そうでなければ変更したファイル一覧）を確認する。
2. **観点チェック**: `review-perspectives` のチェックリスト（仕様適合 / 正しさ / テスト品質 / セキュリティ / 保守性）を実施する。
3. **仕様との突き合わせ**: spec.md の仕様詳細（§ごと）について、どのコード・どのタスクで実装されたかを確認する。実装されていない・仕様と異なる箇所があれば正直に報告する。
4. テスト・Lint・ビルドを実行し、結果をそのまま報告する（失敗を隠さない）。
5. 以下の形式でレビュー資料を提示して**作業を止める**:

   ```markdown
   ## 実装レビュー: NNN-slug

   ### 変更サマリ
   （何をどう実装したか 3〜5 行）

   ### 仕様との対応
   | 仕様 | 実装箇所 | 状態 |
   |---|---|---|
   | §2.1 ... | src/... | ✅ / ⚠️差異あり（内容） |

   ### 検証結果
   - テスト: （実行コマンドと結果）
   - Lint / ビルド: （結果）

   ### 特にレビューしてほしい箇所
   （設計判断した箇所、自信がない箇所、トレードオフ。ファイルパス:行番号 付きで）

   ### 既知の制限・残課題
   ```

6. 「問題なければ `/approve` を実行（または『承認』と返信）してください。修正点があれば指摘してください」と明示する。

## 修正指示を受けたとき

- 軽微な修正: その場で修正し、再度この資料を更新して提示する
- 仕様変更を伴う修正: spec.md を更新（変更履歴に追記）→ 必要なら tasks.md に修正タスクを追加 → 実装 → 再度 /review

## 承認後の後処理

人間が承認したら以下を行う:

1. spec.md の `status` を `done` に更新する
2. **ビジョンへの貢献を記録する**: `dev-docs/vision.md` の「実現の記録」に1行追記する（日付 / ループ名 / 貢献したビジョン項目（§）/ 何がどこまで実現したか）。ビジョン本文（方針）は書き換えない。どの項目にも貢献しないループなら「-」と記録する
3. **現在仕様ドキュメントを再生成する**: /docs の手順で `dev-docs/current-spec.md` を再生成し、このループの PR に含める（ファイルが無ければここで初回生成する）。**再生成の定期タイミングはループ承認後のこの1回だけ**。差分サマリを人間に一言報告する

## 承認後: PR の作成

このループ（spec → tasks → 実装）は `spec/NNN-slug` ブランチ上の1つの PR にまとめる。人間の承認後、リモートリポジトリがあれば PR の作成を提案する:

- タイトル: `[NNN-slug] <機能名>`
- 本文: 上記レビュー資料の内容 + `dev-docs/specs/NNN-slug/spec.md` への参照
- push / `gh pr create` は人間の承認を得てから実行する
