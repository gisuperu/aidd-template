
指摘: （コマンド引数）

`dev-docs/current-spec.md` は /docs がコードとテストから再生成する生成物である。指摘を受けたら**まずコード・テストと突き合わせて**原因を切り分ける。**生成物を直接つぎはぎ修正しない**（更新は常に再生成で行う）。

## 手順

1. **実態確認**: 指摘箇所に対応するコードとテストを特定し、実際の振る舞いを確認する。指摘内容が曖昧なら人間に確認する。
2. **切り分けと対処**（結果を人間に報告してから対処する）:
   - **(a) ドキュメントの抽出ミス**（コードの実態と記述がズレている）: /docs の手順で再生成して修正する。同種のミスが他の箇所にないかも確認し、修正内容を報告する
   - **(b) コードの実態が指摘と違う**（ドキュメントは実態を正しく写しており、人間が望む振る舞いとコードが違う）: ドキュメントは**書き換えない**。これは実装の変更なので、規模に応じて `/check`（軽微な修正を人間が行う）か `/spec`（仕様変更ループ）を提案して停止する
   - **(c) 構成・粒度・表現の改善要望**: 要望を反映して再生成する。以後の生成でも維持すべき要望なら、人間の承認を得て `dev-docs/flows/docs.md` の「生成」手順に追記する（次回以降も同じ構成で生成されるようにする）
3. **手編集の検出**: 人間が current-spec.md を直接編集した形跡があれば、次の再生成で消えることを伝え、その内容を (a)〜(c) のどれかの経路に載せ直す。
4. **報告**: 再生成した場合は frontmatter（`generated:` / `commit:`）を更新し、差分サマリを提示する。

## 位置づけ

- current-spec.md の更新経路は「/review 承認後の定期再生成（1ループ1回）」と「この /docs-fix」の2つだけ。それ以外で再生成しない
- (b) で /check・/spec に進んだ場合、ドキュメントへの反映はそのループ完了時の定期再生成で自然に行われる（先回りして書き換えない）
