# TDD の知識: ケース洗い出しとアンチパターン

## テストケースの洗い出し手法（[RED] タスク作成・実行時）

仕様の各§から機械的に導出する:

1. **同値分割**: 入力を「同じ振る舞いになるグループ」に分け、各グループから1ケース
2. **境界値**: 各範囲の 下限-1 / 下限 / 上限 / 上限+1。コレクションは 0件 / 1件 / 多数。文字列は 空 / 1文字 / 上限 / null/undefined
3. **デシジョンテーブル**: 条件が2つ以上絡む仕様は全組み合わせを表にし、各行1ケース（仕様の分岐表があればそのまま使う）
4. **状態遷移**: 状態を持つ機能は「有効な遷移」と「無効な遷移（エラーになるべき）」の両方
5. **エラーパス**: 外部依存（DB・API・ファイル）ごとに「失敗したらどうなるか」のケース
6. **冪等・重複**: 同じ操作を2回実行するケース

洗い出したケースはタスク本文に列挙する。**仕様に振る舞いが書かれていないケースを発見したら、それは仕様の穴**。テストを勝手に決めず、人間に報告する。

## アンチパターン集（やってはいけない）

| アンチパターン | 症状 | なぜ駄目か / 正しい形 |
|---|---|---|
| 実装詳細のテスト | private関数・内部の中間状態・呼び出し回数をアサート | リファクタで壊れる。公開インターフェースの入出力をテストする |
| トートロジー | 実装と同じロジックで期待値を計算して比較 | 実装が間違っていても通る。期待値はリテラルで書く |
| モック過多 | 依存を全部モックし、モックの戻り値をアサート | モックをテストしているだけ。外部境界（DB・API・時刻）だけモックする |
| 1テスト複数関心事 | 1つのテストで複数の振る舞いをアサート | どれが壊れたか分からない。1テスト1振る舞い |
| テスト間依存 | 前のテストが作ったデータに依存、実行順で結果が変わる | 各テストで setup/teardown。順序非依存に |
| 非決定性（flaky） | 現在時刻・乱数・sleep・実行環境に直接依存 | 時刻・乱数は注入(inject)して固定する |
| アサーション欠如 | 「実行してエラーが出ない」だけのテスト | 期待する出力・状態変化を必ずアサートする |
| [GREEN] でのテスト改変 | 実装を通すためにテストのケース削除・期待値変更・skip | **禁止**。テストが間違いだと思うなら停止して人間に報告 |
| テスト後付けの偽TDD | 実装を書いてから「通るテスト」を書く | 分岐の考慮もれを検出できない。[RED] で失敗を確認してから実装 |
| カバレッジ至上主義 | 行カバレッジを埋めるためだけの無意味なテスト | ケースは仕様の分岐から導出する。カバレッジは結果であって目的ではない |

## 良いテストの型

- **AAA構造**: Arrange（準備）/ Act（実行）/ Assert（検証）を明確に分ける
- **名前**: 「条件_期待結果」が読める名前（例: `残高不足の場合_エラーを返し残高は変化しない`）。テストは実行可能な仕様であり、/docs はこの名前から「現在の仕様」ドキュメントを生成する
- **失敗メッセージ**: 落ちたときに何が期待と違ったか分かるアサートを使う
- **[GREEN] は最小実装**: テストを通す以上のことを書かない。汎用化・先回りは [REFACTOR] か次の [RED] で
