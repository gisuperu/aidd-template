# 仕様書作成の知識

## 機能要件の洗い出しチェックリスト

各機能について、以下を仕様に書いたか（書かない場合は「対象外」と明記したか）:

- [ ] **入力**: 型・形式・必須/任意・値の範囲・上限（文字数、件数、ファイルサイズ）
- [ ] **出力**: 成功時に何が返る/表示される/保存されるか
- [ ] **状態変化**: 何がどう変わるか。副作用（通知・ログ・外部API呼び出し）
- [ ] **権限**: 誰が実行できるか。権限がない場合どうなるか
- [ ] **異常系**: 入力不正・対象が存在しない・外部依存の失敗、それぞれの振る舞い（エラーメッセージ、リトライ、ロールバック）
- [ ] **境界**: 0件・1件・上限ちょうど・上限+1・空文字・null のときの振る舞い
- [ ] **重複・並行**: 同じ操作を2回した場合（冪等性）、同時に実行された場合
- [ ] **既存機能への影響**: 既存の何が変わるか・変わらないか

## 非機能要件のチェックリスト

すべて埋める必要はないが、**触れないなら「今回は対象外」と明記する**（暗黙にしない）:

- [ ] 性能: 目標応答時間・想定データ量・同時利用数（測定可能な数値で）
- [ ] セキュリティ: 認証・認可・入力検証・秘密情報の扱い（security-practices スキル参照）
- [ ] 可用性・信頼性: 失敗時の復旧、データ消失の許容度
- [ ] 運用: ログ（何を出すか）、エラー監視、設定変更の方法
- [ ] 互換性・移行: 既存データ・既存APIとの互換、マイグレーション
- [ ] 国際化・アクセシビリティ

## 仕様書のアンチパターン

| アンチパターン | 例 | 直し方 |
|---|---|---|
| 曖昧語 | 「適切に処理する」「高速に」「〜など」「必要に応じて」 | 具体値・列挙に置換。「300ms以内」「A, B, Cの3種」 |
| 測定不能な要件 | 「使いやすくする」 | 検証手順が書ける形に。「3クリック以内で完了」 |
| 実装の混入 | 「Redisにキャッシュする」を要件として書く | 要件は「2回目以降は100ms以内」、手段は§技術方針へ分離 |
| ハッピーパスのみ | 正常系だけ書いて異常系がない | 異常系・境界を表で列挙（機能要件チェックリスト参照） |
| 暗黙のスコープ | 「やらないこと」が空 | 隣接機能・非機能を挙げて明示的に除外 |
| 未決の隠蔽 | 迷った点を書かずに仮決めで進める | 「未決事項」に仮決め内容と理由を必ず記載 |
| 全部盛り | 1つの spec に複数機能を詰める | ループを分ける。1 spec = 1 PR で収まる規模に |

## 書き方の型

- 各§は **「〜の場合、〜する」** の形式で書く（そのままテストケースに変換できる）。
- 分岐がある仕様は表にする: 条件の組み合わせ × 期待される振る舞い。
- この表が後の [RED] タスクのテストケース一覧の元になることを意識する。
