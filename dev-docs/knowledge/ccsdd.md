# CC-SDD（Claude Code Spec-Driven Development）の原則

このテンプレートのループは CC-SDD を参考にしている。目的は **人間の時間をレビューに集中させ、承認ゲートで品質を担保する**こと。

## 6原則

1. **仕様が唯一の真実**: 実装は仕様に従う。逆ではない。コードと仕様が食い違ったら、仕様を（人間の承認を得て）直してから実装する。
2. **承認ゲートを越えられるのは人間だけ**: draft → approved の遷移は人間の明示的な承認のみ。どれだけ自明に見えても勝手に越えない。
3. **レビューは「判断」を見せる場**: 人間に渡すレビュー依頼には、迷った箇所・仮決めした箇所・トレードオフを必ず明示する。完璧に見せることより、判断ポイントを見せることが目的。
4. **タスクは小さく検証可能に**: 1タスク = 1コミット = 独立して検証できる単位。「検証方法が書けないタスク」は分解が間違っている。
5. **進捗は常にファイルに可視化**: 人間は tasks.md だけを見て状況を把握する。頭の中の進捗とファイルの進捗を乖離させない。
6. **スコープは固定**: 承認された仕様の範囲外はやらない。良かれと思った追加実装が最も信頼を損なう。

## 各フェーズの品質基準

| フェーズ | 成果物が満たすべき基準 |
|---|---|
| /spec | 各§が「〜の場合、〜する」形式でテスト可能。スコープ外と環境変更が明記されている |
| /tasks | 全タスクが spec の§に紐付き、[RED]→[GREEN] 順で、検証方法を持つ |
| /implement | tasks.md が常に実態と一致。仕様外のことをしていない |
| /review | 仕様との対応表に「⚠️差異」を隠していない。テスト結果を加工していない |

## よくある逸脱と正しい対処

| 逸脱しそうな状況 | 正しい対処 |
|---|---|
| 実装中に「ついでにこれも直せる」と気づいた | やらない。/review の「残課題」に記録して人間に提案する |
| 仕様の記述が2通りに解釈できる | 停止して人間に確認。都合の良い方で進めない |
| 仕様どおりだと明らかにバグる | 停止して根拠とともに報告。黙って仕様と違う実装をしない |
| タスクが大きすぎたと実装中に気づいた | tasks.md を分割修正してから続行（分割は報告のみでよい。統合や削除は承認が必要） |
| 人間の返信が承認かどうか曖昧 | 承認か確認する。推定で status を進めない |

## 迷ったときの判断チャート

「これは仕様（現行ループの spec.md）に書いてあるか？」
→ **書いてある**: 実行してよい
→ **書いてない**: 環境変更・スコープ拡大・仕様解釈なら停止して人間へ。純粋な実装詳細（変数名等）なら自分で決めてよい
