# セキュリティ実践知識

初期開発から最高セキュリティで作らないことは多い。そのため各項目を3段階で示す:
**ベスト**（本番の標準）/ **一時回避**（条件付きで許容。選んだら必ず spec の「既知の制限」に**返済条件付きで記録**）/ **禁止**（プロトタイプでも不可）。

## 項目別ガイド

### 秘密情報（APIキー・パスワード・トークン）
- **ベスト**: シークレットマネージャ or 環境変数注入。リポジトリに置かない
- **一時回避**: ローカル開発で `.env` ファイル直書き（条件: `.gitignore` 済み + `.env.example` を用意）
- **禁止**: ソースコード・設定ファイル・ログへのハードコード、コミット履歴に載せること（一度でも載せたら漏洩扱いでローテーション）

### 入力検証
- **ベスト**: 境界（API・フォーム）でスキーマ検証（型・範囲・長さ・許可リスト方式）
- **一時回避**: 内部ツールで信頼できる入力のみなら簡易検証（条件: 外部公開時に必須化）
- **禁止**: ユーザー入力をそのままファイルパス・コマンド・クエリに使う

### SQL / コマンドインジェクション
- **ベスト**: プレースホルダ / ORM。シェルは引数配列で渡す（文字列結合しない）
- **一時回避**: **なし**
- **禁止**: 文字列連結によるクエリ・コマンド組み立て（「自分でエスケープ」も禁止）

### 認証
- **ベスト**: 実績あるライブラリ/IDaaS。パスワードは argon2/bcrypt でハッシュ
- **一時回避**: 開発中は認証モック・固定テストユーザー（条件: 本番ビルドで無効化される仕組みとセット）
- **禁止**: パスワードの平文・可逆暗号保存、自作ハッシュ、認証ロジックの自作暗号

### 認可（権限チェック）
- **ベスト**: サーバー側で毎リクエスト検証。デフォルト拒否
- **一時回避**: 初期は全員同権限の単一ロール（条件: ロール追加時に認可層を入れる前提の設計にしておく）
- **禁止**: クライアント側チェックのみ（UI で隠すだけ）、ID 推測で他人のリソースに届く設計（IDOR）

### XSS / CSRF
- **ベスト**: フレームワークの自動エスケープに乗る。CSRF トークン / SameSite Cookie
- **一時回避**: 認証のない読み取り専用ページでは CSRF 対策省略可
- **禁止**: ユーザー入力を `innerHTML` / `dangerouslySetInnerHTML` 等に未サニタイズで渡す

### CORS / ネットワーク公開
- **ベスト**: 許可オリジンを明示列挙。内部サービスは外部非公開
- **一時回避**: ローカル開発で `*`（条件: 本番設定と分離されていること）
- **禁止**: 認証付き API で本番 `*`、DB ポートのインターネット公開

### 依存ライブラリ
- **ベスト**: バージョン固定（lockfile）+ 既知脆弱性の監査（`npm audit` 等）をフェーズ0で実行
- **一時回避**: 監査で出た低リスク警告の先送り（条件: 既知の制限に列挙）
- **禁止**: 出所不明のパッケージ追加、タイポスクワッティング未確認のままの install（spec 承認時に人間が確認）

### ログ・エラー出力
- **ベスト**: 秘密情報・個人情報をログに出さない。外部向けエラーは詳細を隠し内部ログに詳細
- **一時回避**: 開発中の詳細エラー表示（条件: 本番ではスタックトレース非表示）
- **禁止**: パスワード・トークン・カード番号のログ出力

### レートリミット・DoS 耐性
- **ベスト**: 認証エンドポイント・高コスト処理にレートリミット
- **一時回避**: 初期リリースでは省略可（条件: 既知の制限に記録）

## 運用ルール

1. 一時回避を選ぶときは spec.md の「既知の制限」または「未決事項」に **内容・理由・返済条件（何をトリガーに直すか）** を書き、人間の承認を得る。
2. **禁止**行に該当する実装は、人間に指示されても一度は根拠を添えて確認を求める。
3. /review では一時回避の一覧を必ず「既知の制限・残課題」に含める。
