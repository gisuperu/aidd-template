# AGENTS.md — AI エージェント共通の開発ルール

このファイルはエージェント共通のエントリポイント（Codex / Gemini CLI / Cursor など AGENTS.md を読むエージェント向け。Claude Code は CLAUDE.md 経由で同じ内容を読み込む）。
ルール・手順・知識の**本体はすべて `.aidd-docs/` にあり**、エージェント固有ディレクトリには薄い参照しか置かない。

## テンプレート層とプロジェクト層（DI 的な分離）

このリポジトリのファイルは2層に分かれ、**プロジェクト固有の内容とテンプレート本体を混ぜない**:

- **テンプレート層**（`.aidd-docs/` の rules・flows・knowledge・specs/_template、AGENTS.md、CLAUDE.md、`.claude/` のテンプレート由来ラッパー/ローダー）: ワークフローの仕組み。**プロジェクト固有の仕様・設定・規約を書き込まない**。テンプレート更新時はこの層を丸ごと上書きして取り込む
- **プロジェクト層**（`.aidd-docs/project/`、vision.md、vision-notes/、specs/NNN-*、docs/、README.md）: プロジェクトが所有し、テンプレート更新で上書きされない。固有の内容はここにのみ書く（注入ポイント）

## プロジェクト固有コンテキスト（必読）

プロジェクト概要・スタック・テスト/ビルドコマンド・固有ルールは **`.aidd-docs/project/context.md`** にある。作業前に必ず読み込み、そこに登録されたプロジェクト固有の知識ファイル（`.aidd-docs/project/knowledge/`）も指定されたタイミングで読むこと。

# 開発ワークフロー（最重要ルール）

このプロジェクトは **仕様駆動 + 人間レビューループ** で開発する。
詳細ルールは以下を必ず読んで従うこと:

- `.aidd-docs/rules/workflow.md` — 承認フロー・ステータス管理・ビジョン整合・履歴の扱い
- `.aidd-docs/rules/implementation.md` — TDD・進捗可視化・環境変更のルール

## 絶対に守ること（要約）

1. **人間の承認なしに実装を始めない。** 仕様（spec.md）とタスクリスト（tasks.md）— 設計フェーズを要するループ（spec の `design: required`）では設計書（design.md）も — がすべて `status: approved` になってから実装する。
2. **実装中は必ず tasks.md のチェックボックスを更新する。** タスク着手時に「実装中」マーク、完了時に `[x]`。人間はこのファイルだけを見て進行状況を把握する。
3. **仕様にない変更が必要になったら、作業を止めて人間に報告する。** 勝手に仕様を拡大解釈しない。
4. **テスト駆動で実装する。** タスクは「テスト作成 → 実装 → リファクタ」の順に並んでおり、順序を入れ替えない。テストを弱めて通すことは禁止。
5. **環境を変える操作（`npm install` / `curl` によるダウンロード / ツールのインストール等）は、仕様書に明記され「フェーズ0【人間承認枠】」に載っているものだけを、人間の承認を得ながら実行する。** それ以外の環境変更は一切しない。
6. **現行ループ以外の `.aidd-docs/specs/NNN-*` フォルダは、人間の承認なしに参照・参考にしない。** 過去の仕様書は履歴であり、現在のコードベースと食い違っている可能性がある。現状把握は**コード本体と `.aidd-docs/vision.md`（ビジョン）**から行う。
7. **ビジョン（.aidd-docs/vision.md）は「目指す形」の大域合意。** 作業がビジョンとズレていると気づいたら人間に確認する（spec 側を直すかビジョンを更新するかは人間が決める）。ビジョンを人間の承認なしに書き換えない。
8. **規定された操作は下記の対応する手順書に従い、手順書にない運用を即興で作らない。**
9. **ドキュメントの真実はコードとテスト。** `docs/current-spec.md` は /aidd-docs で再生成する生成物であり、手で編集しない。開発用の資料はすべて隠しディレクトリ `.aidd-docs/` にあり、人間向けの現在仕様ドキュメントだけを `docs/`（ユーザー向けドキュメント領域）に生成する。
10. **各手順書の冒頭に指定された知識ファイル（`.aidd-docs/knowledge/`）を読み込んでから作業する。** 知識ファイルはフェーズごとのチェックリスト・アンチパターン集であり、実行モデル・エージェントによらず品質を揃えるためのもの。
11. **指示を受けたときの既定は主エージェント自身が実作業（調査・実装・執筆）を行う。** 下位モデルの作業者への委任は一つのモードであり、基本は委任しない。委任へ切り替えるのは (a) 委任が明らかに割に合うと判断したとき、または (b) 人間の指示があったとき に限る。委任モードに入ったら主エージェントは統合・裁定・レビューと小さな修正に徹し、作業者のモデルは既定で自身より下位を選ぶ（haiku でこなせる作業なら haiku でよい）。難易度が高ければ自身と同等のモデルを使ってよく、**自身より上位モデルの能力が必要だと判断したら勝手に進めず人間に打診する**（主エージェントは自分でモデルを上げられないため、人間が「上位モデルへ切り替える」か「今のモデルで可能な限り尽くす」かを決める）。並列化そのものは、読み取り専用の調査・検証ならどのフェーズでも自由、書き込みを伴う実装の並列化は `.aidd-docs/rules/implementation.md` の並列規約に従う。**人間との接点（承認・確認・レビュー依頼）と状態ファイル（tasks.md・vision.md・status）の更新・コミットは主エージェントのみ**が行い、委任時はルールの制約を指示文に含める。委任の判断・指示文・検証の詳細は `.aidd-docs/knowledge/subagent-orchestration.md`。

## 操作と手順書の対応

人間が `/aidd-spec 〜` のように操作を指示したら、対応する手順書に従うこと（Claude Code ではスラッシュコマンドが自動で対応する）。表記規約は `.aidd-docs/flows/README.md`。
コマンド名の `aidd-` プレフィックスは、各エージェント組み込みのコマンド（/review /status /docs 等）との衝突を避けるためのもの。手順書ファイル名にはプレフィックスを付けない。

| 操作 | 手順書 | 内容 |
|---|---|---|
| /aidd-vision | .aidd-docs/flows/vision.md | ビジョン（大域方針）の検討・更新 |
| /aidd-spec | .aidd-docs/flows/spec.md | 仕様書ドラフト作成 → レビュー依頼 |
| /aidd-design | .aidd-docs/flows/design.md | 設計書ドラフト作成 → レビュー依頼（spec が `design: required` のループのみ） |
| /aidd-tasks | .aidd-docs/flows/tasks.md | TDD 形式のタスク分解 → レビュー依頼 |
| /aidd-implement | .aidd-docs/flows/implement.md | フェーズ0（人間承認枠）→ 自律実装 |
| /aidd-review | .aidd-docs/flows/review.md | 実装レビュー資料の作成 → レビュー依頼 |
| /aidd-approve | .aidd-docs/flows/approve.md | 承認ゲートの通過（人間が実行） |
| /aidd-check | .aidd-docs/flows/check.md | 人間の細かい修正をレビュー |
| /aidd-status | .aidd-docs/flows/status.md | 進捗・ビジョン実現度の一覧 |
| /aidd-docs | .aidd-docs/flows/docs.md | 現在仕様ドキュメントの再生成 |
| /aidd-docs-fix | .aidd-docs/flows/docs-fix.md | 現在仕様への指摘対応 |
